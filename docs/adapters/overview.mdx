## One resource model, three adapter styles

All adapters share the same `ResourceState<T>` contract:

```ts
type ResourceState<T> = {
  result: T | null;
  loading: boolean;
  error: string | null;
};
```

Your `resources.ts` definitions stay the same when switching adapters. Only the import path and consuming hooks change.

## Feature comparison

| Feature              | Redux                                  | Zustand                                | TanStack Query                                       |
| -------------------- | -------------------------------------- | -------------------------------------- | ---------------------------------------------------- |
| Import path          | `@estedio/react/redux`                 | `@estedio/react/zustand`               | `@estedio/react/tanstack-query`                      |
| Factory              | `createResources` or `createStore`     | `createResources` or `createStore`     | `createResources` or `createStore`                   |
| Hook helper alias    | `createResourceHooks` or `createHooks` | `createResourceHooks` or `createHooks` | `createResourceHooks` or `createHooks`               |
| Provider             | `StoreProvider`                        | Optional `ZustandProvider`             | `TanstackProvider` or your own `QueryClientProvider` |
| Persistence          | Supported                              | Supported                              | Supported                                            |
| Auth token auto-save | Supported                              | Supported                              | Supported                                            |
| Saga effects         | `takeLatest` / `takeEvery`             | Not applicable                         | Not applicable                                       |

## When to use each

<AccordionGroup>
  <Accordion title="Redux">
    Choose Redux when you need:
    - Global store visibility
    - Explicit saga orchestration
    - `takeLatest` or `takeEvery` per resource
    - Easy integration with an existing Redux stack

    [Read the Redux adapter guide](/adapters/redux)

  </Accordion>

  <Accordion title="Zustand">
    Choose Zustand when you need:
    - Minimal state-management overhead
    - Independent resource stores
    - Adapter-managed async state without sagas
    - Optional provider-based config initialization

    [Read the Zustand adapter guide](/adapters/zustand)

  </Accordion>

  <Accordion title="TanStack Query">
    Choose TanStack Query when you need:
    - Query and mutation ergonomics
    - QueryClient-powered server state
    - Existing React Query infrastructure
    - Resource-level persistence layered onto query caching

    [Read the TanStack Query adapter guide](/adapters/tanstack-query)

  </Accordion>
</AccordionGroup>

## Shared infrastructure

All three adapters rely on the same underlying building blocks:

- `ResourceAdapter` for path-param interpolation and HTTP dispatch
- `apiClient` for auth headers, retry, and refresh-token handling
- `localStorageAdapter` and `sessionStorageAdapter` for encrypted persistence
- Shared state behaviors for cache skipping, auth token persistence, and hydration

See [HTTP Client](/core/http-client), [Storage](/core/storage), and [Token Management](/core/token-management) for implementation details.
